## Problem

- Search overlay is **PO-based**: each `SearchableRecord` = one PO entry; index = `build/<lang>/searchindex.pkl.gz` (built by `tools/search/index_builder.py` from `locale/<lang>/LC_MESSAGES/blender_manual.po`).
- **English has no `.po`** → no index → `/api/search?lang=en` returns "No results" (string is in `.rst`/HTML but those are not what the overlay searches).
  - Confirmed: `build/en/searchindex.pkl.gz` was **absent**; fr/ru/vi/zh-hans present (~4 MB gz each).
  - Cause: `serve_docs.py` excludes `en` from the locale scan → `MultiPOWatcher` only finds per-lang PO files → never builds `en`.
- **POT is not auto-fresh:** `locale/blender_manual.pot` is generated by `make gettext`+`make update_po`; no build/live target refreshes it → a POT-based en index would lag the `.rst`. (Rejected; see Alternative.)

## Solution (implemented) — Sphinx extension off the doctree

The build is already a text pipeline: `RST → doctree → resolve → HTML`. Extract searchable messages straight from the doctree — the same source the HTML comes from — exactly as Sphinx's own gettext builder does (`extract_messages` + Catalog-style dedup). Always fresh, incremental, no POT.

### Decision A — extract at `doctree-read`, NOT `doctree-resolved`

Originally specced `doctree-resolved` (as in the reference `translate_po.py`). **Changed after verifying Sphinx internals:** the env is pickled **after read, before write** (`sphinx/builders/__init__.py` — "save the environment" precedes `self.write(...)`). `doctree-resolved` fires during *write*, so per-doc records stored on `env` there are **never persisted** → next incremental rebuild loses every unchanged doc. `doctree-read` fires during *read*, before the pickle → records persist in `environment.pickle` and survive incremental rebuilds. Section ids (anchors) and translatable text are both already present at read time, so nothing is lost.

### Decision B — source language: `msgstr` blank, search `msgid` only

- En records carry `msgstr=""`, `msgstr_stripped=""` (no translation exists).
- `serve_docs._handle_api_search` forces `field=MSGID` when `req.lang == default_language` (via `dataclasses.replace`, since `SearchRequest` is frozen). Belt-and-suspenders: empty msgstr would not match anyway.

### Decision C — record `msgid` + `msgctxt`; locations like `blender_manual.po`

- Extraction yields `(node, msg)` from `extract_messages(doctree)` (skipping substitution definitions, mirroring gettext).
- **Location** = repo-relative RST path + line: `(manual/<...>.rst, node.line)` — matches the PO/parser format (which strips the `../../` prefix). Derived from `node.source` (fallback: `manual/<docname>.rst`).
- **msgctxt**: captured as a field but always `""` — the corpus uses **no** msgctxt (verified: `grep -c "^msgctxt" …vi…po` = 0). Reserved for future use.
- **Dedup** = Sphinx Catalog model: aggregate by `(msgctxt, msgid)`; `locations`/`html_pages`/`section_keys` deduped to unique `(loc, page, anchor)` triples. (One msgid → many location lines, e.g. "Add-ons" in 7 files → 1 record.)
- **Anchor** = nearest enclosing `section.ids[0]` from the doctree — exact, better than the POT-location→`section_map` heuristic.

### Decision D — English-only gate

Extension writes the pickle **only** when `app.config.language in (None, "", "en")` and `app.builder.format == "html"`. Translated langs keep their PO-based watcher index → no double-writer on `build/<lang>/searchindex.pkl.gz`.

## Decision E — server cache must auto-reload the rewritten index (mtime-aware loader)

**Problem.** During `make liveall`, adding/editing an English `.rst` rebuilds the
HTML and rewrites `build/en/searchindex.pkl.gz` (✓), but the **running server
keeps serving stale English search results**. Root cause: `index_loader.load_index`
cached purely by language (`if lang in _cache: return _cache[lang]`) with no
freshness check. The cache is only refreshed by `invalidate_cache(lang)`, which
the `MultiPOWatcher` calls after rebuilding a **PO** index — but **English has no
PO**, so the watcher never watches/invalidates `en`. Net: new English content is
in the pickle and the HTML, but not in search until `serve_docs` restarts. (Same
for *editing* an existing English page, not just adding one.)

> Build side is fine: a newly added RST lands in Sphinx's `added` set; because a
> file was added, globbed-toctree docs are re-read so nav updates; `doctree-read`
> fires for the new doc; orphan (not-in-toctree) docs are still read+indexed with
> a warning. `build-finished` rewrites the full index.

**Fix (chosen).** Make `load_index` **mtime-aware**: stamp `LoadedIndex.mtime`
with the pickle's `st_mtime` at load, and reload when the file's mtime changes.
Self-healing, language-agnostic (also catches PO rebuilds), one `stat()` per
query (negligible). Rejected alternative: extend `MultiPOWatcher` to watch
`build/en/searchindex.pkl.gz` — more moving parts and English-specific.

## Edge cases handled

- **Parallel `-j auto`** → `env-merge-info` + `parallel_*_safe`.
- **Removed/emptied doc** → `env-purge-doc` drops its records; not rewritten.
- **Sections without ids** → anchor `""` (page-level link); never crashes.
- **Nodes without `line`/`source`** → line `-1`; rst-rel falls back to `manual/<docname>.rst`.
- **Substitution definitions / empty messages** → skipped.
- **Schema change later** → bump `env_version` to force a one-time clean rebuild.

## Alternative (rejected) — POT-based index
